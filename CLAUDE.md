# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Dataforms is a **Nextcloud app** (server-side PHP + a Vue 3 SPA) that lets logged-in
users build **registers** (typed, structured data collections), the **forms** that feed
them (with conditional logic), and the **views** that browse them. It is internal /
authenticated-only — no public or anonymous access. See [VISION.md](VISION.md) and
[docs/DECISIONS.md](docs/DECISIONS.md) for scope boundaries (notably: it is *not* a survey
tool, and the REST/OCS API is a first-class contract, not just an SPA backend).

## Commands

```bash
composer install && npm ci   # install deps (PHP 8.1+, Node 20/22)

npm run build                # production frontend bundle into js/ (+ css/)
npm run watch                # rebuild on change during development

make lint                    # all gates: php-cs-fixer + psalm + eslint + stylelint
make test                    # all tests: phpunit + vitest

# Individual gates / suites
composer run cs:check        # php-cs-fixer --dry-run --diff   (cs:fix to apply)
composer run psalm
composer run test:unit       # phpunit, config at tests/phpunit.xml
npm run lint                 # eslint (lint:fix to apply); npm run stylelint for CSS
npm run test                 # vitest run (test:watch for watch mode)

# Run a single test
composer run test:unit -- --filter RuleEvaluatorTest
npm run test -- src/rules/engine.spec.js

npm run test:e2e             # Playwright (needs test:e2e:install once)
```

The built `js/` and `css/` are **git-ignored** — generated by the build, shipped only in
the release tarball. After cloning you must `npm run build` before the app will load.

## Packaging

`make appstore` wipes `js/`+`css/`, rebuilds clean, rsyncs a stripped tree into
`build/sign/dataforms/` (excluding `src/`, `tests/`, tooling, source maps) and prints the
`occ integrity:sign-app` command. The exclude list in the `Makefile` is the source of
truth for what ships — update it when adding top-level dev files.

## Architecture

**Backend** (`lib/`, PSR-4 `OCA\Dataforms\`) follows Nextcloud conventions:
Controller → Service → Db (`QBMapper`). OCS API routes live in
[appinfo/routes.php](appinfo/routes.php) under `/api/v1/...`; the SPA page route is a
`#[FrontpageRoute]` attribute on `PageController`. DI wiring is in
[lib/AppInfo/Application.php](lib/AppInfo/Application.php).

**Frontend** (`src/`) is a Vue 3 + `@nextcloud/vue` SPA. `src/api/*.js` wraps each OCS
resource; `src/components/*.vue` are the builders (SchemaEditor, FormBuilder, RuleBuilder,
AutomationsBuilder, RecordForm/Detail, RecordsView). Built with `@nextcloud/vite-config`.

Three cross-cutting subsystems carry most of the design weight:

1. **EAV storage with typed value columns.** Records store field values in
   `df_record_values`, one row per (record, field), writing exactly one *typed* column
   (`value_string` / `value_number` / `value_datetime` / `value_bool` / `value_file_id` /
   `value_ref_record_id`) — see [lib/Db/RecordValueMapper.php](lib/Db/RecordValueMapper.php)
   (deliberately not a `QBMapper` since the populated column varies). This keeps the schema
   user-definable while staying **portable across MySQL/MariaDB, PostgreSQL and SQLite**.
   Portability is a hard constraint: **no raw SQL, no JSON columns, no engine-specific
   functions** — everything goes through `IQueryBuilder` / migration `ISchemaWrapper`.
   Read [docs/PORTABILITY.md](docs/PORTABILITY.md) before touching the data layer.

2. **Dual rule engine (JS + PHP parity).** Conditional logic (show/hide, require-if,
   set-value, validation, computed) is a *single* rule definition evaluated **both** in the
   browser ([src/rules/engine.js](src/rules/engine.js), `expression.js`) and re-checked on
   the server ([lib/Rules/RuleEvaluator.php](lib/Rules/RuleEvaluator.php),
   `ExpressionEvaluator.php`). The evaluator is a sandboxed, whitelisted expression parser —
   **no `eval`, no dynamic code execution.** When you change rule semantics you must change
   *both* implementations and keep their `*.spec.js` / `*Test.php` suites in lockstep.

3. **Automation engine** (`lib/Workflow/`). On record create/update/delete, when an
   optional condition holds (reuses the rule engine), run an action. Actions are a registry
   ([ActionRegistry.php](lib/Workflow/ActionRegistry.php) + `IAction` implementations) —
   config over code. Cheap internal actions (notify, email, set-field) run inline via
   `AutomationListener`; slow/external ones (webhook, folders, template, calendar, Talk,
   Deck) run in `RunAutomationsJob`. Cross-app provisioning uses a stored **service account**
   (app password) via `NextcloudApiClient`. Note the security-sensitive helpers:
   `PathSafety` (traversal), `WebhookAction` (SSRF guard + HMAC), `ValueInterpolator`.

## Access control

Every endpoint enforces per-register **Read / Write / Manage** server-side (don't rely on
the SPA hiding things). Sharing is modelled in `Db/Share` + `ShareService`. Machine/API
access reuses Nextcloud **app passwords** — there is no custom token system.

## Conventions

- All source files carry `SPDX-License-Identifier: AGPL-3.0-or-later`.
- DB migrations are `lib/Migration/VersionNNNNNNDate*.php`; bump the version in
  `appinfo/info.xml` when shipping schema changes (table prefix `df_`).
- Decisions are recorded append-only in [docs/DECISIONS.md](docs/DECISIONS.md); the API
  contract is in [docs/API.md](docs/API.md); end-user docs in [docs/guides/](docs/guides/).
